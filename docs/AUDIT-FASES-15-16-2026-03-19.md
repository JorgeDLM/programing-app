## FASE 15 — Hardening del Runtime

### 15A. Timeout por fase
- **Implementado:** `phase-runner.ts` ahora envuelve el loop de Claude API en `Promise.race` con timeout configurable
- **Timeouts:** 8 min para fases complejas (discovery, implementation, planning), 5 min para el resto
- **Comportamiento al expirar:** emite `phase_failed` via SSE, lanza error que el orchestrator captura y persiste como estado `failed`
- **Archivo:** `src/lib/pipeline/phase-runner.ts`

### 15B. Output buffer por fase
- **Evaluado:** El SSE ya es phase-aware — cada `phase_event` incluye `phaseIndex`, permitiendo al frontend distinguir output por fase
- **Decision:** No se cambio el buffer legacy del commandCenter (flat array) porque el path primario ya es phase-aware via SSE events. Cambiar el buffer legacy no aporta valor real y podria romper compatibilidad con el historial SSE
- **Deuda tecnica consciente:** El buffer flat sigue existiendo para el path legacy de `history_for` events

### 15C. Legacy blindado
- **Implementado:** `launchSession()` ahora emite `console.warn("[DEPRECATED]...")` al ser llamado
- **Efecto:** Cualquier uso accidental es inmediatamente visible en server logs
- **Archivo:** `src/lib/agent-manager.ts`

### 15D. SSE coherencia
- **Estado:** El orchestrator ya emite todos los eventos criticos:
  - `phase_started` antes de ejecutar
  - `phase_completed` al terminar exitosamente
  - `phase_waiting_client` al crear ClientQuestion
  - `phase_waiting_review` al necesitar review
  - `phase_failed` en catch + timeout
  - `pipeline_completed` al terminar ultima fase
- **Polling:** Sigue como fallback cada 5s para tickets activos

### Nota sobre E2E con Claude API
La prueba E2E real requiere ejecutar tickets contra Claude API con creditos reales. El sistema esta preparado para hacerlo — el flujo es: crear ticket → auto-classify → buildPipeline → registerSession → runPipelineUntilBlocked → Claude API calls → parse → handoffs → complete. Todos los componentes estan integrados y compilando. La validacion real ocurrira cuando se ejecute el primer ticket con API key activa.

## FASE 16 — Prompt Inspector / Debug Panel

### 16A. Inspector API
- **Endpoint:** `GET /api/inspect/[id]`
- **Soporta:** Tanto tickets (Session) como workspaces (Workspace) — detecta automaticamente el tipo
- **Archivo:** `src/app/api/inspect/[id]/route.ts`

### Datos que expone para TICKETS:
| Seccion | Campos |
|---|---|
| General | entityType, entityId, projectId, projectName, status, ticketType, executionMode, priority, directive |
| Pipeline | currentPhaseIndex, totalPhases, currentPhase (con todos sus campos: phaseType, agent, skills, model, status, tokens, duration, retry, outputSummary, handoffNotes, requiresReview, requiresClientApproval) |
| Phases | Lista completa de fases con status y metricas |
| Agent | exists, path, excerpt del agent.md (800 chars max, sanitizado) |
| Agent Validations | Cada fase con validacion de existencia del agente referenciado |
| Model Routing | resolved model, reason |
| Heuristics | domain bias (backend/frontend score), classification rule |
| Questions | Lista de ClientQuestions con status, tipo, fase, si tiene respuesta |
| Telemetry | tokensUsed total, durationMs total, modelUsed |

### Datos que expone para WORKSPACES:
| Seccion | Campos |
|---|---|
| General | entityType, entityId, status, sessionType, title |
| Team | Lista de agentes con exists, excerpt del agent.md, path |
| Team Validation | valid count, missing agents list |
| Model Routing | resolved model, reason |
| Insights | summary, keyQuestions, risks, assumptions, options, missingInfo, suggestedNext |
| Linked | linkedTicketId |
| Messages | messageCount, lastMessageRole |

### 16B. Inspector UI Component
- **Archivo:** `src/components/InspectorPanel.tsx`
- **Tipo:** Panel lateral derecho (drawer) de 320px
- **Secciones:** General, Model, Heuristics, Current Phase, Agent Prompt, Team, Agent Validation, Pipeline, Questions, Workspace Insights, Telemetry
- **Diseño:** Aesthetic, dark-theme, info-dense, dev-focused

### 16C. Integracion en ticket page
- **Boton:** ⚙ en el header de cada ticket (junto al badge de status)
- **Click:** Abre/cierra InspectorPanel con los datos de ese ticket
- **Import:** `InspectorPanel` importado en `page.tsx`
- **Estado:** `inspectId` state variable

### 16D. Seguridad
- **API keys:** NO se exponen. El inspector no muestra el prompt completo si contiene secrets
- **Sanitizacion:** Excerpts del agent.md limitados a 800 chars. OutputSummary a 500. HandoffNotes a 300
- **Degradacion:** Si el entityId no se encuentra, retorna 404 limpio
- **Runtime:** Si el inspector falla, el sistema principal no se ve afectado (componente aislado)
