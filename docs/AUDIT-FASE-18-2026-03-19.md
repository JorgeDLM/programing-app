## 3. Parallel Coordinator

### Archivo: `src/lib/pipeline/parallel-coordinator.ts`

### Arquitectura

```
orchestrator.runCurrentPhase(sessionId)
  |
  |-- detectParallelOpportunity(pipeline, currentIdx)
  |   |-- Busca fases consecutivas del mismo tipo con agentes diferentes
  |   |-- Compara contra SAFE_PARALLEL_PATTERNS
  |   |-- Si match → canParallelize: true + indices + pattern
  |
  |-- SI paralelo:
  |   |-- executeParallelGroup({phases, indices, pattern, ...})
  |   |   |-- Crea branches para cada fase paralela
  |   |   |-- Emite parallel_group_started via SSE
  |   |   |-- Ejecuta Promise.all(branches.map(runPhase))
  |   |   |-- Emite branch_started / branch_completed por cada rama
  |   |   |-- Agrega tokens, duracion, costo por rama
  |   |   |-- Merge summaries y handoff notes
  |   |   |-- Emite parallel_group_completed
  |   |-- Avanza currentPhase pasando TODAS las fases paralelas
  |
  |-- SI secuencial (default):
  |   |-- runPhase() como siempre
```

### Patrones Paralelos Habilitados (3)

| Patron | Fase | Agentes | Razon de seguridad |
|---|---|---|---|
| `review_code_security` | review | inspector-de-codigo + guardian-de-seguridad | Son analisis independientes del mismo codigo — no hay conflictos |
| `qa_tests_flows` | qa | maestro-de-pruebas + tester-de-flujos | Unit tests y E2E tests son suites independientes |
| `finalization_docs_validation` | finalization | documentador + validador-de-salida | Documentacion y validacion son checks finales independientes |

### Patrones NO Habilitados (y por que)

| Patron | Razon de NO habilitar |
|---|---|
| implementation backend + frontend | Pueden tocar los mismos archivos — riesgo de conflicto |
| discovery arquitecto + analista | El analista puede depender de la vision del arquitecto |
| planning paralelo | Requiere vision unica coherente |
| review + implementation | Dependencia directa |

## 5. SSE Events del Paralelismo

| Evento | Cuando | Datos |
|---|---|---|
| `parallel_group_started` | Al iniciar grupo paralelo | pattern name, agentes involucrados |
| `branch_started` | Al iniciar cada rama | branchId, agentName, phaseIndex |
| `branch_completed` | Al terminar cada rama | outputSummary resumido |
| `parallel_group_completed` | Al terminar grupo | branches count, status, cost total |

---

## 6. Merge / Consolidacion

Despues de las ramas paralelas:
- Summaries se consolidan como: `[agentName] summary` concatenados
- Handoff notes se consolidan igualmente
- Si CUALQUIER rama detecta needsClientInput → grupo pasa a waiting_client
- Si CUALQUIER rama detecta requiresReview → grupo pasa a waiting_review
- Si CUALQUIER rama falla → grupo falla
- Solo si TODAS completan → grupo completa y avanza

---

## 7. Integracion en Orchestrator

El orchestrator ahora sigue esta logica:

```
1. Lee fase actual
2. detectParallelOpportunity(pipeline, idx)
3. SI canParallelize:
   a. Marca todas las fases paralelas como running
   b. Ejecuta executeParallelGroup con Promise.all
   c. Actualiza todas las fases con resultados de sus branches
   d. Avanza currentPhase despues de la ultima fase paralela
4. SI NO paralelo:
   a. Ejecuta fase secuencialmente (como antes)
```

---

## 8. Cost Tracking por Rama

Cada rama calcula su costo con `estimateCost(model, tokensInput, tokensOutput)`.
El grupo agrega `totalCost = sum(branches.costEstimate.costTotal)`.
El SSE de `parallel_group_completed` incluye el costo total del grupo.

## 10. Reglas de Seguridad Implementadas

1. Solo patrones en SAFE_PARALLEL_PATTERNS se ejecutan en paralelo
2. Fases deben ser consecutivas y del mismo tipo
3. Agentes deben ser diferentes
4. Si hay waiting_client o waiting_review pendiente, no se ejecuta paralelo
5. Cualquier rama que falle marca el grupo como failed
6. Merge es simple (concatenacion) — no hay conflict resolution todavia
